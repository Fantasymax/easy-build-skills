# 访谈基本功（B1–B10）— skill 全程贯彻的 10 条规则

> **何时读**：Skill 启动时一次性加载本文，作为所有问答交互的"地基"。每条 B 规则在题库 / 形态推断 / lint 等环节都被引用。
>
> **使命**：用户的自由文本回答 + AI 临时追问都需要实时应用这 10 条规则——否则会出现"假数据"流入下游 lint / 形态推断 / 输出模板。
>
> **V0.8 修正**：从 SKILL.md 拆出独立文件（V0.7 仅 B1–B8 内联在 SKILL.md），合并 cowork 的 B7 选项法 + B8 形态推断三铁律，与我们 V0.7 §B8 平台原生 UI 工具规则共存为 B9。

---

## B1 — 行为追问优先于想法追问

❌ 不要问：`你想要 AI 做什么？` / `你希望 AI 是怎样的？` / `你觉得 AI 该…吗？`

✅ 要问：`想想最近一次 [情境] 的具体经过——一步步告诉我那次发生了什么。`

**为什么**：用户对自己的"想法"会美化，对自己的"行为"很难造假。把"想要"逼回"上一次实际做了什么"是反自欺的最强武器。

---

## B2 — 每答必 Why × 1–2

用户给一条回答后，自动追问 1–2 次 `为什么？` 或 `然后呢？`，直到答案落到具体行为或情绪词。

---

## B3 — 偏误自检（每轮回答后扫一遍）

扫**用户的回答**（不是问题），对照清单：

- [ ] 用户用了"我应该 / 我打算 / 一般来说"这类社会期望词？→ 回写："上次实际发生时是怎样的？"
- [ ] 用户给了一字答（是/否/有/没有）？→ 追问让其展开
- [ ] 用户回答与他自己之前的答案矛盾？→ 标矛盾，进入 B4

---

## B4 — 矛盾回写机制（不替用户决定）

遇到矛盾不替用户挑边、不假装没看到。原话回写：

> 你在问题 3 说自己每天写日报；问题 8 说讨厌写日报。请你自己挑一个保留，或写一段说明为什么两者并存。

**这是产品默认行为**，不是 bug。

---

## B5 — 不诱导提问

skill 自带的问题清单文件已经过 4 类偏误 lint（Leading / Shallow / Personal bias / Unconscious bias）。如果 Claude 自己临时追问，遵守同样 4 条规则——详见 `frameworks/lint-rules.md`。

---

## B6 — 题序排列：Game Complexity Arc

题目顺序：暖场（事实）→ 中等（任务回忆）→ 复杂（情绪 / 失败 / 理想态）→ 收尾开放问。所有问题清单文件已按此排序。

---

## B7 — 选项法（设计哲学 3+2+1，呈现受工具硬限制）

> **理论依据**：用户在框架内选择远比自由表达准确——给用户 handle 让他们能反应，而非凭空构造。
>
> **来源**：用户访谈方法论的偏误谱系 + Social Desirability Bias —— 用户自由文字回答会因措辞习惯偏离真实行为，给选项 = 用 recognition 模式降低 self-report bias。

### 7.1 设计哲学：3 强相关 + 2 创意发散 + 1 自定义兜底

每个题目（除非"完全开放型"或"必须贴原物型"）方法论上必须按下面结构组织：

```
{主问}

① {强相关 1 — 最常见或最典型}
② {强相关 2 — 次常见}
③ {强相关 3 — 与 ①② 不同维度的常见}
④ {创意发散 1 — 跳出常规思路的"反向" / "极端" / "类比" 选项}
⑤ {创意发散 2 — 跨界 / 跨领域 / 隐藏需求}
⑥ 其他（请具体描述）：_______
```

### 7.2 呈现层硬约束：AskUserQuestion ≤ 4 option（V0.7 修正）

⚠ Claude Code `AskUserQuestion` 工具硬限制为 **每次调用最多 4 个选项**（自动加 Other），与方法论 3+2+1 = 5+1 冲突。

**调和方案**：
- **简单题**（强相关 ≤ 3）：呈现 3 强 + 1 自定义 = 4 选项，工具自动加 Other = 5
- **复杂题**（要含 2 创意发散）：**拆 2 题**呈现：
  - 第 1 题（≤4 选项）：3 强相关 + Other
  - 第 2 题（≤4 选项）：2 创意发散 + Other（+ 自定义在 Other 里）
- **CLI fallback**（无 AskUserQuestion 工具的平台）：可用 5+1 完整呈现（编号 1–6）

### 7.3 选项设计的硬规则

- ① ② ③ 必须**互相不重叠**且**覆盖典型 80% 场景**
- ④ ⑤ 必须**比 ① ② ③ 更出格**——刺激用户思考"我可能想要的是这种？"
- 选项中**禁止**出现暗示性词汇（"高质量""更好""推荐"）
- ⑥ / Other 永远是开放兜底，**用户输入的自由文字必须保留**

### 7.4 选完之后的追问规则

- 用户选 ① ② ③ → 至少追问 1 次"为什么？" 或 "上次发生时你是这样吗？"（B2 不变）
- 用户选 ④ ⑤ → 追问 "是真的想要这个，还是只是觉得有趣？"——区分"反应"与"承诺"
- 用户选 ⑥ / Other → 自由文字本身可能就是最高价值；但仍要触发 §B3 偏误自检

### 7.5 Schema 校验（合成阶段）

- 用户选了"创意发散"（④⑤）→ 如果与 ① ② ③ 冲突，触发矛盾回写（B4）
- 不得依赖单一题的单一选项做硬决策；必须 ≥ 2 题印证才入 Hard Rules

## B8 — 形态推断与小白友好反馈（V0.4-V0.6 核心）

> 详细规则：见 `frameworks/intent-inference.md`

### 三条铁律

1. **绝不问用户"你要哪种形态"** — 用户是小白，不知道自己想要 atomic / orchestration / with-subagents / plugin-with-hooks / composite-plugin
2. **小白反馈只有 5 选项**：✅ 够用 / 🔼 不够好 / 🔽 太复杂 / ❓ 太简单 / 🖊 自行输入（V0.4.1 ⑤ 出口给熟练用户 agency）— **不要**追问"哪里需要改"
3. **默认偏 Atomic** — 不为展示能力推 Composite Plugin；C4 协作经验"几乎不用"的用户即使触发 hook_signal 也降级 Atomic

### 5 选项反馈映射

- ✅ 够用 → 锁定形态进 Step 6
- 🔼 不够好 → AI 找漏掉的 raw_answer 段加进方案
- 🔽 太复杂 → 自动**降级一级**（Composite → Plugin → Subagents → Orchestration → Atomic）
- ❓ 太简单 → AI 找用户原话里被忽略的高频场景加进来
- 🖊 自行输入（V0.4.1）→ 熟练用户专用：解析用户输入层级（术语级 / 修订级 / 范围级 / 完全 override），按层级处理（详见 intent-inference.md Part 4.2）

### 追加补充

任意选项后都可加 ≤ 50 字的"我还想说…"——AI 据此精细化调整。这同时照顾小白（不强迫追加）和熟练用户（允许精细反馈）。

## B9 — UX 交互呈现（V0.7 — 跨平台兼容）

> **使命**：用户**绝不应该**阅读大段 markdown 题目然后打字回复。必须用平台原生 UI 工具或精简编号选项。
>
> **详细规则**：见 `frameworks/ux-interaction-rules.md`
> **平台映射**：见 `references/cross-platform-tool-mapping.md`

### 五条铁律

1. **首题前必检测** — 按以下优先级查 available tools：
   - `AskUserQuestion`（Claude 全家桶原生）
   - `ask_user_question`（Codex CLI 原生 / MCP 客户端配了 ask-user-questions-mcp）
   - 检测客户端类型（Cursor / Windsurf / Zed / OpenCode 等通过 MCP）
2. **Claude 全家桶必用 AskUserQuestion** — Claude Code 2.1.1+ / Desktop / Claude.ai / Cowork
3. **Codex CLI 有内置 TUI picker** — `ask_user_question` tabbed picker
4. **MCP 客户端通过 ask-user-questions-mcp 支持** — 装 `npx -y auq-mcp-server server` 后等同 AskUserQuestion
5. **CLI fallback 仅用于自部署 LLM / Web ChatGPT 等无 picker 环境** — 精简编号每题 ≤ 10 行

### 单选 vs 多选自动判定

| 题型标识 | 例题 | UI mode 设置 | CLI 提示 |
|---|---|---|---|
| 单选（"选一个最贴近的"）| C1 / A2 / A5 | `multiSelect: false` | "回 1 个数字" |
| 多选（"勾选所有适用的"）| C2 / A3② / A6 | `multiSelect: true` | "回多个数字 1,3,5" |
| 强制 ≥ N | B5 行内规矩（≥3）| 同多选 + 标注 | "至少回 N 个" |

### 反例（在任何平台都不要这样输出）

```
## C1：身份 — 你做什么 + 给谁交付什么

**主问**：

> 用一句话告诉我...

**先选一个最贴近的工作形态**（多选可选 ≤ 2）：

① **文字内容创作**（公众号 / Newsletter / 长文 / 视频脚本 / 课程）
② **软件 / 数字产品**（Indie Hacker / 独立开发 / SaaS）
...
```

→ 用户被迫读 30+ 行才能回——心智成本过高。

### 正例（Anthropic 客户端首选）

```python
AskUserQuestion(questions=[{
    "question": "你做什么 + 给谁交付？",
    "header": "工作形态",
    "multiSelect": False,
    "options": [
        {"label": "文字内容创作", "description": "公众号/Newsletter/视频"},
        {"label": "软件产品", "description": "Indie Hacker/SaaS"},
        {"label": "知识服务", "description": "顾问/教练"},
        {"label": "多重身份并行", "description": "正职+副业+探索"}
    ]  # "其它"由 AskUserQuestion 自动添加
}])
```

### 题库内容如何兼顾两种模式

题库 `question-bank/*.md` 文件保留**详细的"为什么问"+ 追问规则 + 写入字段映射**给 LLM 看；
但 LLM **呈现给用户**的版本必须按本规则精简或调用 UI 工具——
**不要把题库 markdown 原文贴给用户**。

---

## B10 — 用户原话保留（仅 Mode B 强制）

以下内容**禁止 AI 改写 / 润色 / 翻译**，必须逐字保留用户原话：
1. 触发原话片段（"我老板会跟我说……"）
2. 行业行话 / 术语
3. 好范例原物
4. 固定模板（开头结尾、签名）
5. 禁忌原话（"我从来不写……"）

AI 可以改写：流程步骤的连接说明、章节排版、Overview 段、解释"为什么"。
