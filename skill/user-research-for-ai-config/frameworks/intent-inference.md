# 语义反推规则（V0.4.1 核心）

> **角色**：在 Step 5.5（架构反推）由 skill 主流程调用。
>
> **核心原则（V0.4.1）**：**目标用户画像 = 大部分小白 + 少部分熟练用户**。
> - **大部分（小白）** 不知道 single skill / chain / sub-agent / hook / MCP 这些术语 → AI 反推架构 + 用人话呈现 + 用户给 high-level 反馈
> - **少部分（熟练用户）** 知道这些术语 → 在反馈循环留 ⑤ 出口让他们自行 override / 补充
>
> **本 skill 的本质**：AI 通过研究用户反推他需要什么——但**保留熟练用户的 agency**，不强行把所有人当小白。
>
> **绝对禁止**：在**初次访谈环节**让用户回答"你需要 single 还是 chain"这种技术选型问题（V0.4 已删 A11/A12/B11-B14）。
>
> **明确允许**：用户在反馈环节（Step 5.5 ⑤ 出口）**主动**用术语 override AI 的反推。这不是让用户做技术选型，是让熟练用户保留控制权。
>
> **AI 的责任**：从 C1-C4b + A1-A10 + B1-B10 用户对**任务本身**的回答里，扫描语义信号，反推架构方案；翻译成**人话**呈现；接受用户 5 选项反馈；按反馈再调整。

## Part 1 — 语义信号扫描表

> 在用户回答完所有问题（C1-C4b + 模式分支题）后，AI 用下表自动扫一遍。每条信号有"触发模式"（regex 或语义匹配）+ "对应架构装饰"。
>
> **注意**：信号只是**输入**，不是**结论**。最终架构由 Part 2 综合判定。

### 1.1 Skill Orchestration 信号

| 信号名 | 触发模式（在用户回答里搜）| 阈值 |
|---|---|---|
| **多步动词链** | 用户回答里出现"先 X 后 Y" / "X 再 Y" / "第一步 X，第二步 Y" / "完了之后 Z" 模式，且 X / Y / Z 是**不同动词类型** | ≥ 3 个不同任务类型动词 |
| **每步任务类型差异化** | B3 SOP 里的步骤涉及完全不同的认知模式（如"识别"+"判断"+"创作"+"校对"，每个对应不同 mental mode）| ≥ 2 个不同 mental mode |
| **每步用户角色不同** | 用户原话里出现"我那时是 X 角色 / 这一步我是 Y" 切换 | 至少 2 个不同自我定位 |
| **明确串行依赖** | 用户原话里出现"必须先做 X 才能做 Y" / "Y 依赖 X 的输出" | ≥ 1 处明确串行 |

**认定**：≥ 2 个信号触发 → Skill Orchestration 候选

### 1.2 Sub-agent 信号

| 信号名 | 触发模式 | 阈值 |
|---|---|---|
| **多视角并行表述** | 用户原话出现"从 X 角度 + 从 Y 角度" / "同时考虑 X 和 Y 和 Z" / "并行" / "几个维度" | ≥ 3 个独立视角 |
| **视角关注点不重叠** | 各视角能列出**完全不同**的具体关注点（不是一个视角的不同侧面）| 各视角 ≥ 3 条互不重叠的具体项 |
| **审查 vs 执行分离** | 用户原话出现"我做完后 X 来 review" / "出版本 + 审版本" 模式 | ≥ 1 处分离表述 |
| **批量并行处理** | 用户原话出现"30 个客户 / N 份文件，每份独立处理" 且每份**互不影响** | 数量 ≥ 5 + 互独立 |

**认定**：≥ 2 个信号触发 → Sub-agent 候选
**特别注意**："批量"≠"并行"。30 份独立处理可能是 single skill 跑 30 次（用 `/batch`），不一定要 sub-agent。看用户**是否需要每份用不同视角处理**。

### 1.3 Hook 信号

| 信号名 | 触发模式 | 阈值 |
|---|---|---|
| **事件触发表述** | 用户原话出现"每次 X 时" / "X 之后自动 Y" / "提交 / 保存 / 启动 时" | ≥ 1 处明确触发 |
| **硬拦截需求** | 用户原话强调"必须" / "永远不能" / "禁止" + 涉及具体动作（不是泛泛规则）| ≥ 1 处必须级硬规则 |
| **避免遗忘焦虑** | 用户原话出现"我老忘" / "AI 老不记得" / "怕跳过" + 涉及具体边界 | ≥ 1 处遗忘焦虑 |
| **定时 / 周期性** | 用户原话出现"每周 / 每月 / 每天定时" + "希望自动跑"明示 | ≥ 1 处主动表达定时需求 |

**认定**：≥ 1 个**强硬拦截**信号 + ≥ 1 个事件触发信号 → Hook 候选
**反例**：用户只是说"我希望 AI 帮我提醒别忘 X"——这是软提醒（写进 Hard Rules），**不是 hook**。
**反例**：用户在 A12 / B13 风格的题里勾了"每月初定时跑"——但这只是表态，要看是否在 B3 / A1 的真实场景里**主动**提到定时需求。

### 1.4 MCP 信号

| 信号名 | 触发模式 | 阈值 |
|---|---|---|
| **外部系统读取** | C3 工具栈或 B2 输入材料里出现"Notion / 飞书 / Linear / GitHub / Stripe / 网银 / 税务局 / Slack / 企业微信"等外部平台名 | ≥ 1 处 |
| **写回外部系统** | 用户原话主动表达"写回 / 自动发 / 同步到 X" | ≥ 1 处主动写回 |
| **结构化数据查询** | 用户原话提到"读 / 查 / 拉 数据库 / Excel / API" + 频繁使用 | 频次 ≥ 每周 1 次 |

**认定**：
- 仅"读"信号（用户贴内容给 AI，AI 不直接连外部）→ **不需要 MCP**，single skill 能搞定
- "写回 / 自动同步"信号 → MCP 候选（装饰组件）
- 频繁结构化查询 → MCP 候选

**特别注意**：MCP 是**装饰组件**，不是主架构。它增加**用户配置成本**（要装 MCP server）。如果用户只是偶尔需要读外部数据，更稳妥的方案是让用户手动复制粘贴给 AI。

### 1.5 Atomic Skill 默认（绝对锚点）

**Atomic Skill 不需要触发信号**——它是默认。
**保持 Atomic 的强信号**（即使有上面 1.1–1.4 信号，也保持 Atomic）：
- 用户原话强调"重复"而非"复杂"（"每月做 30 次同样的事"）
- 用户表述"一气呵成"心智模式（"我心里是一个完整动作"）
- 用户能把多步表述成"边 X 边 Y"而非"先 X 后 Y"
- 用户的判断节点都在**同一个角色 / 视角**下做出

### 1.6 C 题信号源 — 共享核心 4 题对形态推断的关键贡献（V0.8 新增）

> **来源**：cowork V0.4.1 shape-inference.md 引入 A9.5 协作深度题作为关键信号源。我们 V0.7 没把 C 题作为信号源，导致小白用户即便触发 hook_signal 也按 strong signal 升级 — 这违反"小白默认 Atomic"原则。
>
> V0.8 把 C 题信号显式独立为 §1.6，与 §1.1–§1.5 并列扫描。

| 信号名 | 来源题 | 触发模式 | 对形态推断的影响 |
|---|---|---|---|
| **AI 协作深度极浅** | C4a + C4b 答不上来或都是"几乎不用 AI" | 用户用 AI 经验 < 1 个月，C4a/C4b 都模糊 | **强制降级 Atomic**：即使触发 hook_signal / orchestration_signal，也覆盖回 Atomic（小白用户没有处理复杂架构的能力） |
| **AI 协作深度中等** | C4a 给得清楚 + C4b 模糊 | 用户偶尔用 AI 但很少踩坑 | 不影响主架构判定，正常按 §2.1 跑 |
| **AI 协作深度深** | C4a + C4b 都清楚 + 至少 1 处给出原话片段 | 用户深度用 AI，能精确定位"做对/做错的那句话" | **解锁高级形态**：允许触发非 Atomic 主架构 + 装饰组件叠加 |
| **崩溃修复价值** | C4b 提到 AI 在某场景反复出错 | 用户原话："AI 老是 X 错" / "我每次都得改 Y" | **加权 Hook 装饰**：硬拦截需求 +1 信号 |
| **工具栈极简** | C3 工具栈 ≤ 3 个工具且都是非编码（Word/Excel/微信）| 林女士类型 | **强制降级 Atomic + 不加任何装饰**：用户没有学习配置成本的容忍 |
| **工具栈丰富** | C3 工具栈 ≥ 5 个工具且含 ≥ 2 个编码工具（Cursor/Claude Code/...）| Alex 类型 | 不影响主架构，但 MCP 装饰候选 +1 信号 |

**应用规则**：

```
# 在 §2.1 主架构判定 之前先跑 §1.6 C 题加权
if C4a_C4b_both_blank OR user_ai_experience < 1 month:
    强制覆盖 = 'Atomic'
    跳过 §2.1 升级判定

if C3_tools_minimal_non_coding:
    强制覆盖 = 'Atomic'
    跳过所有装饰组件判定（MCP/Hook 都不加）
    理由：林女士 dogfood 验证 — 用户主动反对配置成本，即使 B14 提 Excel 也不加 MCP

if C4b_repeated_failure_pattern:
    hook_signals_strong += 1  # 加权
```

**为什么放在 §1.6 而非 §1.1–§1.5**：C 题信号是**元层级**的（关于用户能力 vs 工具复杂度的匹配度），不是关于"任务本身需要什么形态"的。先扫元层级，再扫任务层级。

## Part 2 — 架构综合判定（V0.6 关键修正）

> 把 Part 1 的信号汇总后，**分两步独立判定**：先定主架构，再判定每个装饰组件。这两步**不耦合**——避免 V0.4.1 时期"信号互相干扰导致直接推 hybrid"的问题（Mode A 三角对比验证：3/3 都是 Single 主架构 + Alex 单独触发 Hook 装饰，符合 V0.6 拆分逻辑）。

### 2.1 步骤 1：主架构判定（互斥，必选 1 个）

> **核心原则**：默认 Single；只在用户原话有**强复杂需求信号**且**通过反例校验**时才升级到非 Single。

```
# 默认锚点
主架构 = Atomic Skill

# 升级条件（每个分支都需要 atomic_anchor < 3 才考虑升级）
if orchestration_signals >= 2 AND atomic_anchor_signals < 3:
    主架构 = Skill Orchestration

elif subagent_signals >= 2 AND atomic_anchor_signals < 3:
    主架构 = Skill with Subagents

elif hook_signals_strong >= 1 AND user_explicitly_wants_event_driven AND atomic_anchor < 3:
    主架构 = Plugin with Hooks

else:
    主架构 = Atomic Skill   ← 绝对默认（V0.5 数据：90%+ 用户走这条；Mode A 三角 3/3 验证）

# 注：不在主架构判定阶段考虑 Composite Plugin。Composite 只在用户 dogfood 已证 atomic 装不下后升级（V0.5 composite-plugin/README.md 明示）
```

### 2.2 步骤 2：装饰组件判定（可叠加，与主架构独立）

> **关键变化（V0.6）**：装饰组件**不再依赖主架构判定结果**——独立扫信号决定。这避免了 V0.4.1 时期"主架构是 hook 就不再考虑 MCP 装饰"等耦合错误。

```
装饰组件 = []  # 独立扫，可叠加

# MCP 装饰
if Part 1.4 MCP 信号触发 AND 用户主动表达"省得我手动复制粘贴" AND 不反对增加配置成本:
    装饰组件.append('MCP')
# 注：林女士虽然 B14 提到 Excel/网银，但她说"用不来那种东西"反对配置成本 → MCP 不加

# Hook 装饰（如 hook 不是主架构）
if Part 1.3 hook_signals_strong >= 1 AND 主架构 != Hook:
    装饰组件.append('Hook')
# 注：Alex 主架构是 Single，但 git diff 涉及敏感目录的硬拦截需求 → Hook 装饰必加

# Skill Orchestration 装饰（如 chain 不是主架构）
if 出现 1 个明显的 phase 分界但不到 chain 级:
    装饰组件.append('Phase 分界提示')

# 装饰判定的"不加"原则（V0.6 强化）
- MCP 增加用户配置成本 → 必须用户主动想要 + 不反对手动复制粘贴 + 频次 ≥ 每周 1 次才加
- Hook 增加 settings.json 改动成本 → 必须有"硬拦截"信号（不是软提醒）才加
- 装饰组件 ≠ "更全更好"——每个装饰都有成本，宁缺勿滥
```

### 2.3 装饰组件成本表（V0.6 新增）

| 装饰 | 用户成本 | 触发要求 |
|---|---|---|
| **MCP** | 装 server / 配凭据 / 学新工具 | 用户**主动**表达"想 AI 直连外部" + 频次高 + 不反对配置 |
| **Hook** | 改 settings.json / chmod 脚本 / 学 hook 概念 | 用户**主动**表达"必须硬拦截，软提醒不够" + 1 个具体事件 |
| **Phase 分界提示** | 几乎无成本（仍在单一 SKILL.md 内）| 1 个明显的 phase 分界但不到 chain 级 |

### 2.3 反例（强制保 Single）

即使有上述信号，也强制 Single 的情况：

1. **C2 任务清单显示重复频次极高**（每周 ≥ 5 次）+ 用户原话表述"一样的事重复做" → 几乎必 Single（用 examples/ + assets/templates/ 复用）
2. **用户在 A2 选了"实习生 / 助理"角色比喻** → 通常意味着 single + 简单流程
3. **用户在 A4 的"动词 + 对象"是单个清晰动词**（如"整理 / 抽取"）+ 一个明确对象 → Single
4. **用户没主动用"先 / 后 / 第一步 / 然后 / 同时 / 并行 / 自动 / 每次"等流程词** → Single

## Part 3 — 翻译成人话（呈现给用户）

> AI 反推完架构后，**永远不要直接说"我推荐 Skill Orchestration"**。用户不知道这是什么。

### 3.1 通用呈现结构（V0.5 关键修正）

**所有架构的方案都按下面 5 段结构呈现，让用户 1 屏看完整不漏内容**。这避免了林女士 dogfood 中暴露的问题（轮 1 方案把红线藏在"它不会做..."描述里，用户感觉"漏了"）。

#### 6 段结构（V0.6 — 加段 3.5 机械化检查点）

```
段 1：核心动作（一句话）
─────────────────────────────────────
"你以后 [触发场景] 时，跟 AI 说 '[B7 触发原话]'，
 它会 [B1 一句话定义动词] + [输出物]。"

段 2：✅ AI 自动会做的事（具体到步骤）
─────────────────────────────────────
- [Do 栏 + B3 SOP 关键步骤 转化成人话]
- ...

段 3：❌ AI 永远不会做的事（你的红线，AI 自动拦着自己）   ← V0.5 关键独立段
─────────────────────────────────────
- [A8 红线 第 1 条 转化]
- [A7 Don't 栏关键条 转化]
- [B5 关键 Anti-pattern 转化]
（这些是 AI 自动拦自己的硬规则，不需要你每次提醒）

段 3.5：⚙ 机械化检查点（V0.6 新增）— AI 不靠"思考"就能判定的硬规则
─────────────────────────────────────
> 不是所有规则都靠模型决策。下面这些 AI 跑完 grep / regex / 字符串匹配就知道结果，
> 比"模型判断"准 10 倍：
- [Alex 例：git diff 涉及 migrations/ api/ stripe/ 任一目录 → 必停下问]
- [林女士例：新会话第一句必须是 "这次处理哪家客户" → 阻塞等待]
- [小马例：检测段落含 "✨" 或 "💪" emoji → 自动删除]

段 4：⏸ AI 拿不准会停下来问你的（语义判断）
─────────────────────────────────────
- [A7 Ask 栏第 1 条]
- [B4 判断节点关键条]
- [A9 escalation 触发条件]

段 5：📁 交付物文件清单（+ 配置成本）
─────────────────────────────────────
- 主文件：[1 / N 个，描述]
- 支持文件：[examples / references / templates / hooks / agents 按架构]
- 配置成本：[如有，必须明确告知用户需要手动做什么]
```

#### 关键原则（V0.6 修订）

1. **段 3"AI 不会做"必须独立显眼** — 不藏在"它不会写..."这种否定式描述里。用户应该 1 屏看到自己的红线已经被 AI 内化为硬规则。
2. **段 3 推荐 ≥ 3 条但不强制（V0.6 修订）** — 优先扫 A7 Don't / A8 红线 / B5 Anti-pattern。如果用户原话**真的红线很少**（< 3 条），不强行凑数：段 3 标"红线很少 — 你的工作风险面比较小，挺好"+ 列出实际数量。"红线少"是弱信号 = 工作类型本身风险面小（如纯创作展示 / 数据汇总），不是 AI 没扫够。
3. **段 5 必须告知配置成本** — hook 要改 settings / MCP 要装 server 必须明确说"你需要手动 X"。不要藏。
4. **段 3.5 机械化检查点（V0.6 新增）** — AI 反推时**主动尝试**把段 3 的红线压缩为可机械化的检查（grep / regex / 字符串 match / 阻塞 ASK）。3 角色 dogfood 共性：用户最痛的红线常常都能压缩成机械化检查（Alex 的目录匹配 / 林女士的新会话阻塞 / 小马的 emoji regex）。机械化 > 语义判断（前者准 10 倍）。如果一条红线**完全无法**机械化，保留在段 3 不挪到段 3.5；段 3.5 只放真能机械化的。

#### 各架构差异（仅段 1 + 段 5 不同）+ ASCII 文件树预览（V0.8 新增）

> 段 5 呈现给用户时，**必须附 ASCII 树**让用户 1 屏看清"我会拿到几个文件"。

**Atomic Skill**：
- 段 1：单步触发，一句话给 AI 即可
- 段 5：1 个主文件 SKILL.md + 支持文件；零配置成本（拷文件夹即可）
- ASCII：

```
{skill-name}/
├── SKILL.md           ← 主文件（一句话触发）
├── examples/
│   ├── good-1.md      ← 你写过的好范例
│   └── bad-1.md       ← 反例（AI 不能这样做）
└── references/
    └── glossary.md    ← 你的术语表
```

**Skill Orchestration**：
- 段 1：分 N 步走，每步做完给你看，你说继续才下一步
- 段 5：1 个总入口 + N 个分步文件；零配置成本
- ASCII：

```
{skill-name}/
├── SKILL.md                  ← 总入口（编排 N 步）
└── sub-skills/
    ├── 01-{step-1}/SKILL.md  ← 分步 1
    ├── 02-{step-2}/SKILL.md  ← 分步 2
    └── 03-{step-3}/SKILL.md  ← 分步 3
```

**Skill with Subagents**：
- 段 1：让 N 个"分身"同时从 N 个角度看一遍，最后汇总一份给你
- 段 5：1 个汇总入口 + N 个分身文件；零配置成本
- ASCII：

```
{skill-name}/
├── SKILL.md                ← 汇总入口（调度 N 个分身）
└── agents/
    ├── {perspective-1}.md  ← 分身 1（如 "从安全角度"）
    ├── {perspective-2}.md  ← 分身 2（如 "从性能角度"）
    └── {perspective-3}.md  ← 分身 3（如 "从可读性角度"）
```

**Plugin with Hooks**：
- 段 1：每次你 [事件] 时，AI 会自动 [硬拦截动作]，即使你忘了让它做
- 段 5：1 个主文件 + **1 个配置片段（你需要手动复制到一个叫 settings.json 的文件里，这是这种方案的额外成本）** — 必须明确告知
- ASCII：

```
{plugin-name}/
├── .claude-plugin/
│   └── plugin.json         ← plugin 元数据
├── skills/
│   └── {skill-name}/SKILL.md
└── hooks/
    └── pre-{event}.sh       ← 事件触发自动拦截
+ settings.json 配置片段（复制到你的全局 settings）  ← 额外配置成本
```

**Composite Plugin**：
- 段 1：组合上述（按主架构 + 装饰组件描述）
- 段 5：N 个文件 + 可能含配置片段；按组合复杂度告知成本
- ASCII：

```
{plugin-name}/
├── .claude-plugin/
│   └── plugin.json
├── skills/                  ← 主功能
│   ├── {main-skill}/SKILL.md
│   └── {sub-skill}/SKILL.md
├── agents/                  ← 专精分身
│   └── {agent}.md
└── hooks/                   ← 自动化触发
    └── pre-{event}.sh
+ settings.json 配置片段
```

**装饰组件追加**：
- MCP 装饰：段 1 加 "AI 会顺便帮你接上 Notion / 网银，省得你每次复制粘贴"；段 5 加 "需要装一个小工具（叫 MCP server），我会告诉你怎么装"
- Hook 装饰（如不是主架构）：段 1 + 段 5 同上述 Plugin with Hooks 细节

#### V0.5 之前的旧模板（V0.4.1）— 已废弃

旧的 1 段散文式呈现（"我建议这样做：你只要一句话告诉 AI..."）容易让用户感觉"AI 不会做"被一笔带过。V0.5 改 5 段独立结构后，林女士 dogfood 的轮 1 ④ 反馈应该不会再发生（红线本就独立段，用户一屏看到）。

### 3.2 不准用的术语

呈现方案给用户时，**禁止**出现这些词：
- ❌ "single skill" / "skill chain" / "sub-agent" / "hook" / "MCP" / "frontmatter" / "permissionDecision" / "context: fork"
- ❌ "调用 / 派发 / 分发"
- ❌ "上下文隔离 / fork / parallel"
- ❌ "settings.json" 直接出现（除非用户已经在工具里看过这个名字）

**可以用**：
- ✅ "1 个文件 / N 个文件"
- ✅ "AI 会先做 X 再做 Y"
- ✅ "并行" / "同时几个角度"（只在用户已用过这种描述时）
- ✅ "自动 / 每次 X 时"
- ✅ "守门员 / 助手 / 工具"

## Part 4 — 用户 high-level 反馈循环

> 呈现方案后，让用户用极简反馈，AI 反推怎么调。

### 4.1 反馈选项（5 个，V0.4.1 关键修正）

```
┌─ 这个方案怎么样？ ─────────────────────────┐
│                                            │
│  ① 够用，就这样                            │
│  ② 不够好，[一句话哪里没覆盖]              │
│  ③ 太复杂，让我手动的事太多 / 步骤太多      │
│  ④ 太简单，漏了我经常需要的 X              │
│  ⑤ 我自己来调整：[自行输入想怎么改]         │
│                                            │
└────────────────────────────────────────────┘
```

**⑤ 自行输入出口的设计意义**（V0.4.1 关键）：
- 大部分用户是小白，靠 ①②③④ 已经够用
- 但 **少部分是 AI 熟练用户**，他们可能：
  - 知道 Claude Code 能力，想直接说"加一个 sub-agent 做 review"
  - 觉得 AI 给的选项不准确，想基于 AI 选项**补充修订**
  - 知道自己要什么架构，想直接 override（如"不要 chain，给我 single + 多写 examples"）
- ⑤ 是这类用户的出口；保留他们的 agency。**没有 ⑤ 就把这部分用户挡在门外**——他们会觉得这个 skill 太"傻瓜"。

### 4.2 AI 反推规则

**用户选 ①** → 完成，进入 Step 6 输出文件

**用户选 ②（不够好）** → AI 扫用户给的"哪里没覆盖"那句话，对照原 raw_answers 里**没用进当前方案**的回答段，找出可能漏的：
- 检查：B5 行内规矩有没有都进 Hard Rules？
- 检查：B8 好范例 / 差范例有没有进 examples/？
- 检查：B7 触发原话有没有进 description？
- 把找到的"漏掉的部分"加进方案，重呈现

**用户选 ③（太复杂）** → AI 反推**简化方向**：
- 如果当前是 Chain → 合并相邻步骤（把任务类型最相近的两步合一）
- 如果当前是 Sub-agent → 减少视角（保留最强的那 1-2 个）
- 如果当前是 Hook → 改成软提醒（写进 Hard Rules，不用 hook）
- 如果当前是 Hybrid → 拆装饰，回主架构 V1
- 重呈现，告诉用户："改成这样，你的手动操作只剩 [X 个动作]，比之前少 [Y]"

**用户选 ④（太简单）** → AI 反推**该补什么**：
- 用户给"漏了 X" → 在 raw_answers 里搜 X 相关的原话回答（哪些题答到了 X）→ 把这部分加进方案
- 如果用户讲不清楚漏了什么，AI 反问 1 个具体问题：「我看你之前说过 [B 某题原话片段]——这部分是不是我没体现进方案？」

**用户选 ⑤（自行输入调整指示，V0.4.1 新增）** → AI 解析用户输入的**层级**，分类处理：

| 用户输入层级 | 识别信号 | AI 处理 |
|---|---|---|
| **术语级**（熟练用户）| 用户使用 single skill / chain / sub-agent / hook / MCP / context: fork 等术语 | **跳过反推**，直接按用户指定的架构产出。AI 只确认细节（"你要的 sub-agent 应该有哪几个视角？"）|
| **修订级**（基于 AI 方案补充）| 用户引用方案某部分（"把第 3 步" / "去掉 review" / "加一个守门员"）| 在原方案上做局部修改，不重跑反推 |
| **范围级**（更细的 high-level）| 用户给比 ②③④ 更具体的人话约束（"我要自动跑但只在 git push 时" / "只对长文档生效"）| 把约束注入对应组件（hook matcher / paths 字段等），重呈现 |
| **完全 override**（小白也能做）| 用户说"我不要这个方案，我自己写" + 给出他的版本 | AI 转为"按用户指示落地"模式，不再尝试反推；用户成为方案主宰 |

**关键规则**：
- 选 ⑤ 后**不要质疑用户**。即使他用了术语，AI 不要说"你确定要 sub-agent 吗？这个比较复杂"。这是在 alienate 熟练用户。
- 选 ⑤ 后呈现新方案时**保留人话** + 用户原话片段 — 这样如果他后悔可以 ② / ③ / ④ 反馈。
- 选 ⑤ 仍然是循环的一部分（用户调整完后看新方案不满意可以再 ②③④⑤）—— 直到 ① 才停。

### 4.3 循环上限

最多反馈 3 轮：
- 第 1 轮：用户给 ①②③④
- 第 2 轮：AI 调整后用户再反馈
- 第 3 轮：如还没收敛，AI 主动说「我们目前的版本可能已经够用——建议先用 V1 跑一次真实任务（dogfood），看哪里出问题，再针对那个问题升级 V2。这比纸面继续打磨更高效」

## Part 5 — Part 6：与既有规则的衔接

### 5.1 与 lint Layer 1 §B4 矛盾回写的关系

仍然适用。如果用户 ① 说"够用"但 AI 反推发现用户某个回答**自相矛盾**（如 B11 里 A 说要"全自动"但 B 说"每步停下问"），AI 仍然要回写，让用户自己仲裁——但**不是问"你要 single 还是 chain"，而是回写**用户自己的两段原话**让用户挑。

### 5.2 与 question-bank V0.4 的关系

**V0.4 关键变化**：删除 A11 / A12 / B11 / B12 / B13 / B14 这种"让小白选架构"的题。**所有架构判定都在本文件由 AI 反推**。

剩下的 question-bank 仍然有效（C1-C4b（C4 V0.8 拆为 C4a/C4b）+ A1-A10 - {A11,A12} + B1-B10 - {B11,B12,B13,B14}）。

### 5.3 与 5 种架构模板的关系

5 种模板仍在 `templates/mode-b/`。但用户**不需要选**——AI 反推后**自动**拷对应模板，按 B1-B10 字段填进去。

## Part 6 — 边界（V0.4 设计意图重申）

> 这个 skill 服务的是"**不懂如何设计 AI 工作流的人**"。这是它存在的全部理由。
>
> 一旦让用户做技术选型——"single 还是 chain？" / "要不要 hook？" / "MCP 还是 scripts？"——这个 skill 就退化成"专业用户的问卷"，而专业用户根本不需要这个 skill。
>
> **本文件的所有 lint 规则、信号扫描、综合判定**都是为了让 AI 替小白用户做技术决策——但**让用户做最终的、人话的、business-level 的确认**（够 / 不够 / 太复杂 / 太简单）。
